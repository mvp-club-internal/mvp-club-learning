01

What is an LLM?

a prediction machine · the engine inside every agent

An LLM is a prediction machine. Trained on enormous datasets, it does one thing: given some input, predict the most likely next output. That's the entire engine. What changes from chatbot to agent is the range of things we let it predict.

"The sky is…"
→
LLM
→
blue89%
cloudy5%
falling3%
The "harness" — what we let the LLM predict
1 Chat
You type → LLM → predicts text backThe original interaction layer. ChatGPT in 2022.
Input is text. The only output it can predict is more text.
↓ expand the prediction space ↓
2 Chat + tools
You type → LLM → predicts text OR a tool callread a file · search the web · call an API · run code
Same LLM. We just gave it more options for what to predict next.
↓ expand further ↓
3 Agent
You give a goal → LLM → predicts the next step, in a looptext · tool · observe result · ↻ adjust · ↻ another tool · text · done
Same prediction engine, just looping with a much bigger output space. That's an agent.
The LLM doesn't change. What changes is the harness around it — the menu of choices we let it predict. The next section is about that third row.

Skills

the modern unified primitive · Claude triggers them

A directory at .claude/skills/<name>/SKILL.md. The description field is the matchmaker — Claude scans descriptions to decide which skill to run when you ask something. You never name the skill.

"make this rhyme: I love coffee in the morning"
tldrSummarizes long text into bullets.
rhyme-itRewrites a sentence to rhyme.
translateTranslates between languages.
Claude scans descriptions → finds the match → runs that skill.
Three frontmatter flavors: default (both invoke) · disable-model-invocation: true (you only — what "commands" used to be) · user-invocable: false (Claude only). Follows the Agent Skills open standard.

Subagents working together

a real workflow · main Claude orchestrating three subagents
A skill is…
A playbook Claude loads into its own context when something matches. Guidance Claude follows while staying in your conversation.
vs
A subagent is…
A worker Claude dispatches to a separate fresh context. It does the job alone and returns only the result. Defined by an agent.md file.

One subagent is useful. Several subagents working in concert is where decomposition pays off. You give a goal; main Claude breaks it into pieces and dispatches each piece to a fresh-context worker. Your main conversation only ever sees the summaries.

YOU
"Write me a blog post on prompt engineering for beginners."
↓goal
MAIN CLAUDE
Decomposes: research → draft → critique. Dispatches three subagents via the Task tool, each with their own brief.
↓ ↓ ↓dispatch via Task tool
Researcher
.claude/agents/researcher.md
Reads 10+ sources on prompt engineering, ranks the best 5, summarizes each.
↳ Returns: 5-bullet summary. The 10+ raw articles stay in its own context.
Writer
.claude/agents/writer.md
Drafts a 600-word post in a friendly, beginner-aimed voice from the researcher's summary.
↳ Returns: a clean draft. Drafting iterations stay in its own context.
Critic
.claude/agents/critic.md
Reviews the draft for accuracy, jargon, and beginner-friendliness.
↳ Returns: revision notes. The line-by-line scan stays in its own context.
↑ ↑ ↑only summaries return
MAIN CLAUDE
Has: a 5-bullet research summary, a draft, and revision notes. Applies the notes to the draft. Produces the final post.
↓polished post
YOU
Receive only the finished post. The 10+ sources, the rough draft, the back-and-forth — none of it cluttered your thread.
What's happening: three agent.md files in .claude/agents/ defined three workers. Main Claude (acting on your behalf) decomposed the goal and dispatched each worker via the Task tool. Each ran in its own fresh context, did its job, returned a summary. Your main thread stayed clean. This is the pattern that makes complex workflows tractable.

01

Decompose the workflow

a workflow → a list of tasks → a team of agents

Before any building, the real skill: breaking a workflow into agents. Start with the job as a human would describe it, split it into tasks where the work genuinely changes, then map each task to one agent.

The workflow

Take a topic + audience and produce a polished, ready-to-post LinkedIn post.

→
The tasks
  • Research the topic
  • Write a draft
  • Edit & refine
→
The agents
  • post-researcher
  • post-writer
  • post-editor
The boundary rule: split into a new agent where both the job changes and the context it needs changes. Research needs the open web; writing needs a voice; editing needs quality criteria. Three different jobs, three different heads → three agents. Don't over-split (one agent per sentence) or under-split (one agent doing everything).
This is exactly what Anthropic means by "each consideration handled by a separate call." One job per agent keeps each one focused — and keeps your main thread clean.
02

The handoff contract

the most important design decision in the whole system

Agents don't share a brain. Each runs in its own fresh context and only sees what it's handed. So how does the writer know what the researcher found? A file. Each agent writes a file the next one reads. The output format of one agent is the input spec for the next — that's the contract.

⚙

How one agent runs a crew

the mechanism · before we build, a look under the hood

We keep saying "the command dispatches three agents." Mechanically, what is that? No magic — it's the same loop you met in Part 1, and dispatching an agent is just a tool call.

1 · The agent loop (Part 1, recap)

Claude Code runs what the docs call the agentic loop. The model predicts the next step — text or a tool call — the tool runs, its result feeds back into the model's context, and the loop repeats until the job is done.

Predict
Decide next step
The model predicts text or a tool call.
→
Act
Run the tool
Read a file, search, run a command — or dispatch an agent.
→
Observe
Feed result back
The tool's result re-enters the model's context.
↻ repeat until the task is done

2 · Three forces are doing the work

Here's the part worth slowing down for. When you run /draft-post, three different kinds of thing combine — and only one of them is actual machinery. Keep these colors in mind:

Convention
Names and folders you made up. post-researcher, the pipeline/ folder, the section headings in the brief. Claude Code attaches no special meaning to these — they're your choices, written in plain English.
Mechanism
The few things the program does automatically: load files from .claude/, turn /draft-post into a prompt, and give each subagent a fresh context when the Agent tool is called.
Claude decides
The model reading your instructions and choosing what to do — including when to call the Agent tool and what to put in each handoff file. This is judgment, not a rule.

3 · Watch the containers of action

Here's what's really happening when you see "post-researcher" pop up in your chat. Everything lives inside Claude Code, the program (the big frame). It hands files to a context window — a bounded space where one mind works. Watch files fly out of your directory into whichever window is live, and new files get written back. When a subagent is dispatched, a fresh context window opens, does its job, and closes — its memory gone.

So when a subagent appears in your chat, this is the moment: Claude Code opened a brand-new context window, loaded only that agent's file plus the task Main Claude wrote, and let it work in isolation. Whatever it does inside never touches your main chat — only its final result comes back.
Does Main Claude check the subagent's work? Only if you told it to. Our skill just says "run the next one," so Main Claude passes the baton without grading the handoff. Want a quality gate? Add it to the skill — "if the draft is weak, send it back" — and Main Claude will. Evaluation is a convention you write, not something the machinery does for you.
The payoff: almost none of this is special "orchestration machinery." Claude Code only does two mechanical things — load your files and give each subagent a fresh context window. Everything else is your plain-English convention plus Claude's judgment in reading it. That's why you can build a whole crew with nothing but markdown files.
One hard rule (this part is mechanical): a subagent can't dispatch its own subagents — the Agent tool isn't available inside one. So the orchestrator must be the top-level session (our skill), with the three workers one level below. That's exactly the shape we're about to build.

Build the editor

the quality gate · judgment, not just text

The editor is the last line of defense. It reads the draft, measures it against explicit criteria, fixes what's weak, and writes the final post. Spell out "good" — if the editor's bar is vague, bad posts sail through.

Make criteria explicit: length, hook, one takeaway, no fluff, no generic AI phrasing, an engaging close. The clearer the bar, the better the gate.

Wire the crew

one skill that runs the whole line

Now the conductor. We make a skill — .claude/skills/draft-post/SKILL.md — whose whole job is to run the three agents in order. Typing /draft-post fires the pipeline end to end.

The skill is the prompt; Claude is the orchestrator. The skill doesn't "call" agents itself — its instructions tell Claude to dispatch each subagent in sequence, and Claude does the dispatching. disable-model-invocation means it only fires when you type the command — predictable for a live run.
Heads up: new .claude/agents/*.md files load at session start. If /draft-post can't find an agent, restart Claude Code so it picks them up. (Skills hot-reload; agents don't.)

Do it live · draft-post skill

---
name: draft-post
description: Run the full content pipeline — research, write, edit — into a finished LinkedIn post.
disable-model-invocation: true
argument-hint: "[topic] [audience]"
---

or edit yourself — dispatch each subagent in order:

1. Use the post-researcher subagent with the topic

2. Use the post-writer subagent. It reads the brief
   and writes pipeline/draft.md.
3. Use the post-editor subagent. It reads the draft
   and writes pipeline/final_post.md.

07

Break it on purpose

where the lesson actually lands

A working pipeline is satisfying. A broken one is where you learn. Let's sabotage the contract and watch the chain stumble — so you can feel why the handoff format matters.

Try it · snap the contract

  1. Open .claude/agents/post-researcher.md and delete the output section headings (the ## Key Points / ## Audience Insight / ## Hook Angle block). Leave it as just "research the topic."
  2. Re-run /draft-post on the same topic.
  3. Open pipeline/research_brief.md, then pipeline/draft.md. Watch how a shapeless brief produces a mushier draft.
What you just proved: the agents are only as good as the contracts between them. The researcher's output format is the writer's input spec. Vague handoff → the whole chain degrades. Tighten the brief's structure and the draft sharpens right back up.
This is the one idea to carry out the door: when a multi-agent system misbehaves, check the handoffs first. Open the files between the agents — the break is almost always there.
