Phases 1–5 today · Phases 6–7 in Week 3: Build

Vocab for today. PRD (Product Requirements Document): the spec, or what you're building toward this session. User story: one sentence about what a person wants to do and why. Scope: what's in v1. Out of scope: what isn't, which matters just as much.
01

Problem

What are we solving, for whom, and why?

You already worked this out last week. Now you're distilling it. You pull the core problem out of your intent file and state it clearly in two or three sentences.

Wardrobe app · MVP Club's example

"When I see someone wearing an outfit I love, I have no easy way to find pieces that actually look like it. Searching by hand is slow, and what I turn up is usually close in name but not in look, or it costs way more than I want to spend on the whole outfit."

Phase 1 · Problem

  1. Open Claude Code in your project folder so it can read your intent.md. It stays available for all five phases.
  2. Start with the first prompt to open things up. The second one asks Claude where your problem description has gaps.
  3. Use the third when you're ready to lock in your problem statement. It starts your PRD file, so the doc is building from phase one.
Read my intent.md and tell me in one sentence what you think the core problem is.
What's unclear or underspecified about how I've described the problem?
Write a 2–3 sentence problem statement and start a new file, docs/prd.md, with it under a 'Problem' heading.
02

Users

Who shows up to use this, and what do they need?

User stories make a vague idea concrete. Each one is a sentence about who wants to do what, and why. Three to five is the right number. If you're writing eight, your scope is already too big.

Wardrobe app · MVP Club's example

Single user: Jill. Uses saved outfit photos from Instagram, magazines, screenshots. Sets a budget for the whole outfit. Wants near-identical matches she can actually buy, not a flood of options.

  • As Jill, I want to upload or paste an outfit photo so I can find the pieces in it.
  • As Jill, I want to set a single price range so every match stays within my budget.
  • As Jill, I want to see 1–2 close matches per item so I have clear choices, not noise.
  • As Jill, I want each look auto-saved to my library so I can come back and buy later.

Phase 2 · Users

  1. Claude Code already has your intent.md from the project folder, so you don't need to add it again.
  2. This phase is a conversation. Talk with Claude about who really uses this and what they need before anyone writes a single user story. Tell it where it's wrong, and let it ask you questions back.
  3. Once you've talked it through, the last prompt turns the discussion into your final list and adds it to your PRD.
Based on my intent.md, who do you think actually uses this app? Let's talk it through before we write anything down.
Here's who I think uses it and why: [your thoughts]. Where am I wrong, and what am I not thinking about?
Now that we've talked it through, turn it into 3–5 user stories and add them to docs/prd.md under a 'Users' heading. Flag any that feel redundant or out of scope.
03

Scope

What's in v1, and what's explicitly not

This is a critical phase. A lot of builds get too big and become hard to manage when we try to fit everything into the scope of an MVP or V1. The way through is to be disciplined about what's in v1 and honest about what isn't.

The out-of-scope list is the more important one. It's how you actually finish. When in doubt, put it out of scope. You can always add it back in Week 3.
Wardrobe app · MVP Club's example

In v1: photo upload or paste · price range slider · outfit analysis · 1–2 product matches per item within budget · look page with Buy links · auto-save to library

Out of scope: per-item budgets · login or accounts · cross-device sync · mobile app · paid shopping API · refresh stale matches

Phase 3 · Scope

  1. The first prompt asks Claude for the smallest version that still works. It forces a real choice.
  2. The second builds your out-of-scope list. Push Claude to be aggressive here. Longer is better.
  3. The third is a scope check. Run it before you leave this phase.
  4. The last one saves both lists into your PRD.
What's the smallest version of this that still solves the core problem?
Give me an out-of-scope list, and make it at least as long as the feature list.
What am I most likely to try to add mid-build that we just said is out of scope?
Add our in-scope and out-of-scope lists to docs/prd.md under a 'Scope' heading.
04

Tech

Where does this live, and how is it built?

You don't need to know anything about tech to run this phase. Claude asks you questions you can answer from your own experience, and it makes the technical calls based on what you tell it.

What Claude will ask you: Where should this run? Should your data still be there if you close the browser? Does anyone else need to access it? You describe what you need, and Claude picks the stack.
Wardrobe app · MVP Club's example

Stack: Next.js. It runs in the browser, with one small server route to hold the API key.

Data: the library is stored inside the browser (IndexedDB), so there's no database to set up. It works the same locally or deployed to Vercel.

Hosting: local first, with the option to deploy to Vercel later for a real URL.

Trade-off accepted: the library is tied to the browser it lives in, which is good enough for a personal tool.

Tech terms · what they do for your build

You don't need to memorize these. They're here so Claude's recommendations make sense. Each one is framed by what it does for you and what it costs.

Front end
The part people see and click: your screens, buttons, and layout.
For your build: this is where your time goes on look and feel. More polish means more iterating, but it's also what makes the app feel like yours.
Like a restaurant: the dining room, what guests see and touch.
Back end
The part that runs behind the scenes, where the real work happens and where secrets like API keys are kept.
For your build: you need one as soon as your app uses a paid service or handles anything you don't want sitting out in the open. It's one more piece to run and host.
Like a restaurant: the kitchen, where the food actually gets made. Guests never go back there.
Node.js
The engine your app's behind-the-scenes code runs on. Most beginner web apps use it.
For your build: you install it once at the start, and after that it just quietly runs your app while you work. Claude handles the rest.
Like a restaurant: the gas and electricity the kitchen runs on. Turn it on once and everything works.
Database
Where your app's data is kept so it's still there after you close the tab.
For your build: add one when your data needs to survive a refresh or follow you across devices. Skip it and you can store data right in the browser instead, simpler, but tied to that one computer. That's the trade-off our wardrobe app made.
Like a restaurant: the walk-in fridge, where things are kept so they're still there tomorrow.
API keys
A password that lets your app use a paid service, like an AI model.
For your build: it has to live in the back end, never the front end. Leave it exposed and anyone can grab it and run up your bill.
Like a restaurant: the company card behind the bar. Great for serving the premium stuff, but leave it on the counter and anyone runs up the tab.
Language
What your code is written in, like Python or TypeScript. Same idea as different spoken languages.
For your build: Claude picks it, and you rarely need to care which one.
Like a restaurant: the language the recipes are written in. English or French, same dish.
Framework
A starter kit so you're not building from scratch, like Next.js or Angular.
For your build: it saves you a huge amount of time, but you build things its way.
Like a restaurant: a meal-kit. The structure's done, you just finish it their way.
Packages
Ready-made pieces of code you drop in for a feature, like an image uploader or a calendar.
For your build: each one adds something useful and one more thing your app now leans on.
Like a restaurant: appliances you add. A blender lets you make smoothies, but it's one more machine to maintain.

Phase 4 · Tech

  1. The first prompt tells Claude to ask you questions before it decides anything. Answer in plain language.
  2. The second gets you a recommendation with the trade-offs explained in plain terms. Push back if anything feels off.
  3. The third captures the stack and constraints and adds them to your PRD.
Before recommending anything, ask me the questions you need answered.
Recommend the simplest option. Explain the trade-offs in plain language.
What constraints should I write down so we don't forget them during the build? Add the stack and constraints to docs/prd.md under a 'Tech' heading.
05

PRD

It's been building all session. Read it as one, then check it

You've done the hard thinking, and your PRD has been building the whole way. Each phase wrote its piece into docs/prd.md, so the document already exists. It's the single source of truth for your build. A solid PRD doesn't guarantee you get exactly what you pictured, but it gets you building something much closer to it from the start, instead of way off base and facing a lot of rework. Next week Claude Code reads it straight from your project folder so it knows what you're going for.

What a PRD contains: problem · users · user stories · core features (v1 only) · out-of-scope list · tech stack and constraints. One screen. Short is better.

Read it as one, then run 3 checks

  1. Does it hold together? Read the whole PRD start to finish. This catches gaps between phases, repeats, and anything vague.
  2. What's true for the user? Looks at the finished app from your user's side.
  3. What's true for you as the builder? Deploying it, running it, what it costs, and keeping it going.
Read docs/prd.md back to me as one document, and flag anything that's missing, repeated, or contradictory.
If we build exactly what's in this PRD, what will be true for the user?
And what will be true for me as the builder? Think about deploying it, running it, what it costs, and keeping it going over time.
Good enough beats perfect. If your checks come back lining up with what you want for the user and for yourself, the PRD is ready. You're not chasing a flawless document, just one solid enough to build from. You'll keep refining as you go next week.
It's already saved: because you built it phase by phase, your PRD already lives in docs/prd.md. That file travels with the project, and Claude Code reads it automatically in every build session.

Before next week

Turn your PRD into a build plan, then get set up safely

Your PRD says what you're building. The last step today turns it into a build tracker, a plain checklist of the steps to get there, so next week you work straight off it instead of staring at a blank screen.

Make your build tracker

Ask Claude to turn your PRD into a step-by-step build plan, then save it next to your PRD so it travels with the project.

Using my PRD, write a build tracker: a checklist of the steps to build this, in order, that I can work through and check off. Save it to docs/build-tracker.md.
How you'll use it: next week you'll open your project folder, open the tracker, and see exactly where you are. Pull it up every time you sit down with Claude so you always know the next step.
Make a safety copy first. Before you build, copy your whole project folder (Cmd-C, then Cmd-V) and build in the copy. If a build session gets messy, your clean PRD and tracker are still sitting safe in the original, ready to start fresh from.
